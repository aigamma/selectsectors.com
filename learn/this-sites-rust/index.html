A tour of this site's Rust.

Every code block on this page is inlined from crates/backtest-core/src/ at build time via Vite's ?raw import. When the Rust source changes, this page changes on the next deploy. There is no separate copy of the code to drift out of sync.

The crate at a glance

The crate has one job: take a bar series and a strategy spec, return a backtest result. The implementation splits into four concerns:

  • bars.rs defines the input type (one daily OHLC bar plus volume).
  • metrics.rs holds the pure-function performance metrics (Sharpe, drawdown, CAGR, hit rate, equity curve).
  • strategies/ is one module per strategy, each exposing a positions(bars, params) function, plus a StrategyKind enum that dispatches by variant.
  • error.rs holds the hand-rolled error enum.
  • lib.rs ties everything together and exposes the #[wasm_bindgen] entry point that the Netlify function calls.

The WASM entry point

Everything starts here. The JS side calls run_backtest({ symbol, bars, strategy }); the Rust side deserializes the input, runs the strategy, and returns a serializable result struct. The #[wasm_bindgen] attribute is the only WASM- specific code in the function body; the rest is plain Rust.

crates/backtest-core/src/lib.rs

Strategy dispatch via tagged enum

The StrategyKind enum is the dispatch table. Each variant either has no payload (BuyAndHold) or wraps a strategy-specific Params struct. The positions method matches on self and forwards to the right submodule. This is the idiomatic Rust replacement for a virtual dispatch table; no dyn Trait, no allocation, no runtime lookup.

The #[serde(rename_all = "snake_case")] attribute tells serde to encode BuyAndHold as "buy_and_hold" on the wire, which is the convention the JS dispatcher expects.

crates/backtest-core/src/strategies/mod.rs

A concrete strategy: SMA crossover

Each strategy is a small module that takes bars: &[DailyBar] and a Params struct and returns a Result<Vec<f64>, BacktestError>. The vector has the same length as the input bar series; each entry is the strategy's desired exposure at the close of that bar (1.0 long, 0.0 flat, -1.0 short for a future short-capable strategy).

Three patterns worth noticing in the SMA implementation:

  1. Parameter validation up front. Bad params return InvalidParam errors rather than producing nonsense output. The cost of the check is negligible; the benefit is the caller sees a clean error message instead of a NaN-filled result.
  2. Length check next. A 50-bar SMA needs at least 50 bars before it can compute its first value. We check and return NotEnoughBars rather than silently producing zeros for the warm-up period.
  3. Sliding-window mean. The rolling_mean helper adds the new value and subtracts the value leaving the window each step. O(n) total work for n bars regardless of window size, vs O(n*w) if we recomputed the mean from scratch each step.

crates/backtest-core/src/strategies/sma_crossover.rs

Pure-function metrics

The metrics module is the simplest part of the crate to read and the easiest to test. Each function takes a slice of f64 and returns a single f64 (or a Vec<f64> for the equity curve). No state, no I/O, no error paths beyond the empty-input degenerate case. This is the kind of code where Rust's strictness costs you nothing because there is nothing tricky to be strict about.

crates/backtest-core/src/metrics.rs

The no-lookahead engine

The hardest part of writing a correct backtest is making sure your strategy cannot see the future. The naive bug is using close[i] in a signal that determines position[i] — the position at bar i is now determined by information that includes bar i. In live trading you would only learn that close after the bar ended; you cannot have already taken that position.

We solve this at the engine layer with apply_positions_to_bars: the position at the close of bar i determines the exposure during bar i+1's price change. Individual strategies cannot accidentally cheat because they never see the realized return; they only set the position vector and the engine lags it.

crates/backtest-core/src/strategies/mod.rs (excerpt)

Why this is the right shape. The lag is centralized in one function. Every strategy in the crate gets the no-lookahead guarantee for free, and a future strategy author cannot accidentally violate it without rewriting the engine itself. Invariants like this are best enforced at the layer that owns them, not asked of every caller.

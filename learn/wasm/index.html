Lesson 06 · Rust to WASM

Rust to WebAssembly.

Compiling a Rust crate to WebAssembly is one wasm-pack build away. The interesting parts are the JS-Rust boundary: how #[wasm_bindgen] marks the exports, how serde-wasm-bindgen marshals JSON-shaped data, and what wasm-pack build --target nodejs actually drops into pkg/.

The compilation target

Rust compiles to many targets. For WebAssembly, the two most common are:

  • wasm32-unknown-unknown – bare WASM, no OS, no stdlib. The default for wasm-pack output.
  • wasm32-wasi – WASM with the WASI system interface, used for server-side WASM runtimes.

This site uses wasm32-unknown-unknown: the backtest math has no I/O needs (all inputs come in as a JS object, all outputs go back as a JS object), so the unknown-unknown target is the smallest and fastest to load.

What wasm-pack build --target nodejs produces

The --target nodejs flag tells wasm-pack to emit JS glue suitable for a Node.js runtime. Running: 

wasm-pack build crates/backtest-core --target nodejs --out-dir ../../pkg --out-name backtest_core

drops these files into pkg/:

  • backtest_core_bg.wasm – the actual WebAssembly binary (~98 KB for this site's crate).
  • backtest_core.js – the JS glue that instantiates the WASM module and exposes the #[wasm_bindgen] functions as CommonJS exports.
  • backtest_core.d.ts – TypeScript declarations for the exports.
  • package.json – a package manifest that lets npm-style consumers pick up the JS as the main entry.

The Netlify background function imports it with a normal ES module import (Node 22 supports ESM-importing CJS modules) and calls run_backtest directly.

The #[wasm_bindgen] attribute

The attribute marks a function (or struct, or method) as callable from JS. The wasm-bindgen macro generates a thin trampoline that handles the JS-WASM type marshalling: 

#[wasm_bindgen]
pub fn run_backtest(input: JsValue) -> Result<JsValue, JsValue> {
    let inputs: BacktestInputs = serde_wasm_bindgen::from_value(input)
        .map_err(|e| JsValue::from_str(&format!("bad input: {e}")))?;
    let result = run_backtest_inner(inputs)
        .map_err(|e| JsValue::from_str(&e.to_string()))?;
    serde_wasm_bindgen::to_value(&result)
        .map_err(|e| JsValue::from_str(&format!("serialize: {e}")))
}

The signature is JsValue -> Result<JsValue, JsValue> because JsValue is the opaque "anything from JS" type. Errors come back as strings the JS caller sees as thrown exceptions.

serde-wasm-bindgen does the marshalling

The actual JS-to-Rust conversion happens in serde_wasm_bindgen::from_value: it walks the JS object using serde's Deserialize trait and populates the corresponding Rust struct field by field. Same in reverse via to_value for the output.

The benefit of going through serde instead of hand-writing the JsValue manipulation is that Rust's type system stays in charge. Our BacktestInputs struct has #[derive(Deserialize)]; the input shape is enforced at the boundary, and a malformed input (missing field, wrong type) produces a clean error message instead of a runtime crash inside the strategy code.

The "bridge cost"

There's a real cost to crossing the JS-WASM boundary: serde-wasm-bindgen walks the entire input object on every call (~tens of microseconds for ~1000 bars), and walks the entire output object on the way back. For short-running functions called many times, this can dominate.

Two ways to mitigate:

  • Batch. Call WASM once with the full input rather than many times with pieces of it. The backtest crate does this: one call per backtest, not one call per bar.
  • Typed arrays. For large numeric arrays, you can pass a Float64Array directly via wasm-bindgen without going through serde; the JS-side Float64Array backs onto WASM linear memory with no copy. We don't use this here because the bar series isn't large enough for the optimization to matter; for a much bigger workload it's the right next step.

Loading WASM in the Netlify function

The wasm-pack output uses fs.readFileSync at module-load time to read the .wasm bytes from disk and instantiate the module. That works fine inside a Netlify function as long as the .wasm file is shipped alongside the function bundle. Netlify's bundler does not statically trace the readFileSync call, so we have to tell it explicitly which files to include: 

[functions."backtest-background"]
  included_files = ["pkg/**"]
  node_bundler = "esbuild"

With that block in netlify.toml, the entire pkg/ tree ships into the function bundle and the readFileSync resolves at runtime.

The takeaway. wasm-pack handles 90% of the Rust-to-WASM packaging. wasm-bindgen marks the exports; serde-wasm-bindgen does the JSON marshalling; the JS glue knows how to instantiate the module. The remaining 10% is bundler config to ship the .wasm file alongside the consuming code. Once wired, calling Rust from a Netlify function is indistinguishable from calling any other CommonJS module.

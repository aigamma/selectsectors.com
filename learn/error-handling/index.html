Lesson 05 · Error handling

Error handling without thiserror.

Rust has no exceptions. Every function that can fail returns Result<T, E>, and the caller must either handle the error or propagate it explicitly with ?. The question is what to use for E. This site uses a hand-rolled enum; here is why, and when the popular thiserror macro becomes the better choice.

Result is the only failure mode

A Rust function can return T (always succeeds), Option<T> (might return nothing, with no failure detail), or Result<T, E> (might fail, with the failure shape E). The ? operator propagates an Err up the call stack; the compiler enforces that the calling function either also returns Result (with a compatible E) or handles the error before returning.

The convention for the error type is to define an enum with one variant per failure mode the function can have. This gives callers exhaustive pattern matching on the failure cases, which is the same compile-time-checked dispatch we saw with StrategyKind.

BacktestError, the actual enum

crates/backtest-core/src/error.rs

Four variants, each describing a specific failure mode the backtest can produce. The Display impl formats each variant into a human-readable string the JS side can show to the user. The std::error::Error impl marks the type as a real Rust error type for any third-party code that wants to treat it as one.

Why not just use thiserror?

The popular thiserror crate is a derive macro that generates the Display and Error impls automatically from format-string attributes on each variant. Most Rust crates use it. The equivalent BacktestError with thiserror is: 

#[derive(Debug, Clone, thiserror::Error)]
pub enum BacktestError {
    #[error("not enough bars: strategy requires at least {required} but only {given} were provided")]
    NotEnoughBars { required: usize, given: usize },
    // ... etc
}

Less code, identical behavior. The reason we don't use it here is bundle size: thiserror brings in ~60 KB of expanded macro output for a four-variant enum we can write by hand in fifty lines. The crate compiles to WebAssembly and ships inside a Netlify function bundle; every dependency costs bytes.

That trade-off shifts as the error surface grows. At ten variants, the hand-rolled version becomes tedious and error-prone (literally; you might forget to add a match arm to Display for a new variant). At that point thiserror is the right pick. The crate's own comment notes the migration condition.

The ? operator

The ? operator is the syntactic sugar that makes error propagation ergonomic. It applies to any Result<T, E> and either unwraps the Ok value or returns the Err from the enclosing function: 

fn run() -> Result<f64, BacktestError> {
    let positions = strategy.positions(&bars)?;
    //                                          ^ on Err, return immediately
    let sharpe = compute_sharpe(&positions);
    Ok(sharpe)
}

The expression before ? must return a Result whose E is convertible (via From) to the enclosing function's error type. For BacktestError we don't need any From impls because the strategies and the run_backtest function all share the same error type.

What about anyhow?

anyhow::Error is a type-erased error wrapper: you put any error in, and the caller can only look at the string representation. It's the right pick for application code where the caller doesn't care which specific failure occurred ("just print something useful and exit"). It's the wrong pick for library code, where callers do care and need to pattern-match on the specific error kind.

This site's crate is library code (consumed by the Netlify function), so the typed enum approach wins. The JS side could also use the string representation (BacktestError implements Display via to_string()) but it can also key off the variant tag if it wanted to.

The pattern. Library code: hand-rolled enum at small variant counts; thiserror once the count grows. Application code where callers don't need structured access to failure detail: anyhow. The lazy default for an exploratory script: anyhow. The thoughtful default for a crate other code depends on: typed enum.

Lesson 04 · Enums & dispatch

Enums as dispatch tables.

Rust's enum is much more than a list of constants. It is a tagged union: each variant can carry its own data, and pattern matching on the variant lets you do type-safe, exhaustive dispatch with no runtime overhead. This site's StrategyKind is a working example.

The basic shape

An enum lists its variants. Variants can be:

  • Unit: BuyAndHold (no payload)
  • Tuple: SmaCrossover(Params) (positional payload)
  • Struct: Limit { price: f64, qty: u32 } (named-field payload)

The Rust compiler knows the complete set of variants at compile time. A match expression on the enum must handle every variant or the compiler rejects it; this is the "exhaustiveness" guarantee.

The dispatch pattern

This site's StrategyKind uses the tagged enum to dispatch the right strategy at runtime without virtual calls or HashMap lookups. The positions method matches on self and forwards to the right submodule:

crates/backtest-core/src/strategies/mod.rs (StrategyKind)

Three things to notice. First, the enum variants name the strategy types directly; there's no separate registry to maintain. Second, the positions method takes &self (borrowed) and pattern-matches; the compiler turns this into a jump table at compile time. Third, adding a new strategy means adding a variant to the enum and a new arm to the match; the compiler will refuse to build until you handle the new variant in name() and positions() both.

How serde encodes it on the wire

The #[serde(rename_all = "snake_case")] attribute on the enum controls JSON encoding. With this attribute, the externally-tagged serialization (which is serde's default for enums) produces: 

// BuyAndHold (unit variant)
"buy_and_hold"

// SmaCrossover(Params { fast: 20, slow: 50 }) (tuple variant)
{ "sma_crossover": { "fast": 20, "slow": 50 } }

The JS dispatcher in the backtest function builds these shapes from the {name, params} shape the frontend sends. Round-tripping through serde produces a fully-typed Rust enum on the other side; the compiler knows which variant it is and the strategy-specific Params struct is accessible by pattern destructuring.

vs trait objects (dyn Trait)

An alternative dispatch pattern in Rust is the trait object: define a trait Strategy with a positions(&self, bars: &[DailyBar]) method, implement it for each strategy type, store strategy instances as Box<dyn Strategy>, dispatch via virtual call.

Trait objects are the right pick when the set of implementations is open (third-party crates can implement your trait) or when you need to store many instances of different types in the same collection (Vec<Box<dyn Strategy>>). They have a small runtime cost: each call is one indirection through the vtable.

The enum approach is the right pick when the set is closed and known at compile time (exactly the case here: five strategies, all in this crate, all known to the JS caller). The compiler can optimize the match better than the runtime can optimize the vtable lookup.

vs function pointers / HashMaps

You could also model dispatch with a HashMap from name to function pointer: 

let dispatch: HashMap<&str, fn(&[DailyBar]) -> Vec<f64>> =
    [("buy_and_hold", buy_and_hold::positions),
     ("momentum", momentum::positions)]
    .into_iter().collect();

This works but breaks the type safety of the per-strategy Params: each function pointer would have a different signature for params, so you can't put them in the same map without erasing the type. The tagged enum keeps the type system on your side: each variant has its specific payload type, the compiler knows it, and the dispatch is exhaustive by construction.

When the enum gets too big

If the set of variants grows past ~20, the positions match arms become tedious and the enum file itself becomes unwieldy. At that point the right pattern is often to combine: an enum for the "kind" tag that dispatches into trait-object-based per-kind handlers. But for 5 strategies in a small crate, the pure enum is the simplest correct shape.

The rule of thumb. Closed set, known at compile time, few variants: tagged enum. Open set (anyone can add a new implementation), or many heterogeneous instances stored together: trait object. Performance- critical hot path: enum (zero indirection) over trait object (one vtable lookup per call).

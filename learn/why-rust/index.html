Lesson 01 · Why Rust

Why Rust?

Rust is not the right tool for every problem. It is the right tool for some of them. This site's backtest math is one of those problems, and the four reasons it picked Rust over the plausible alternatives are the same four reasons most teams that pick Rust pick it.

The four reasons

1. Native-speed math without a runtime tax

The dominant cost of a backtest is walking the bar series and computing rolling statistics. For our universe a single strategy on a 1000-bar SPX history is a few hundred thousand floating-point operations. Done in TypeScript with for loops over typed arrays, that takes ~5-20 ms because V8 has to dynamically dispatch every operation and check types at runtime. Done in Rust compiled to WebAssembly, the same workload takes ~0.5-2 ms because the loop is monomorphized at compile time and the inner ops become tight scalar code the WASM engine can vectorize.

The ~10x speedup matters because a strategy library scaling to hundreds of parameter combinations across 23 symbols and 5 years of bars wants to run on one CPU core in seconds, not minutes. Rust gets us there with margin to spare.

2. Predictable memory behavior

The bar series for a typical backtest is 1000-2000 doubles (40-160 KB). In Rust we allocate it once, walk it in-place, and free it deterministically when the request handler returns. There is no garbage collector to fire on a request we care about, no second-call latency spike because the GC decided this was the moment to compact.

For a Netlify function with a 26-second synchronous budget (and 15 minutes for the background variant), determinism is worth more than peak throughput. A 2-ms backtest that always takes 2 ms is more useful than a 1-ms backtest that occasionally takes 200 ms because a GC pause fired in the middle.

3. Compile-time errors that catch real bugs

Rust's type system catches a specific class of bugs that dynamically-typed languages catch at runtime if at all: accidentally indexing past the end of a slice, accidentally passing the wrong argument to a function that expected a different shape, accidentally returning None from a function whose caller assumed Some. In a numerical context where the inputs come from JSON and the outputs go back to JSON, having the compiler check the shape end-to-end means we ship fewer "backtest returned NaN" bugs.

The trade-off is real: Rust will reject code that another language would accept, and you have to rewrite the code until the compiler is happy. The first month of writing Rust is mostly an argument with the borrow checker. The second month is mostly winning those arguments. After that, the friction is replaced by a quiet confidence that the code does what you think it does.

4. Clean WebAssembly output

Rust is one of the few languages with first-class WebAssembly tooling. wasm-pack build --target nodejs takes our crate and produces a 98 KB .wasm file plus ~16 KB of JS glue that the Netlify function can require() directly. No additional runtime, no standard library that has to be re-implemented in WASM, no glue layer that interprets the language at runtime.

The alternative path (write the math in Python and have the function call Python via a sidecar) would add a process boundary, multiple seconds of cold start, and a complete extra runtime for what amounts to a few hundred lines of numerical code. Rust + wasm-pack ships the entire engine as two files the function can load in ~20 ms.

When NOT to reach for Rust

The reasons above are also the cases where Rust is overkill:

  • Glue code, I/O orchestration, request routing. The dispatcher at /api/backtest is TypeScript because it spends 99% of its time waiting on Supabase, the cache lookup, and the background-function dispatch. Rust would buy nothing here and would cost the simpler tooling story TypeScript has on Netlify Functions.
  • Frontend logic that does not have tight math. The chart rendering, the form handling, the chat panel: all vanilla TypeScript. Bringing Rust to the frontend for non-math work pays the WASM bundle cost (60-100 KB uncompressed) for no measurable benefit.
  • Prototypes and scripts. A one-off Python script that downloads the Massive grouped endpoint and writes a CSV is fine in Python. Rust here is the wrong point on the iteration-speed-vs-runtime-speed curve.
  • Code that does not have to run fast. If the workload is small enough that even a 10x speedup is invisible to the user, the speedup is not worth the cost of writing the workload in Rust.
The decision rule. Reach for Rust when the code is in a hot path (the user notices the difference), is numerical or systems-y (the type system pays off), and will be touched by multiple developers over time (the compile- time guarantees compound). Reach for TypeScript or Python otherwise.

What this means for the site

The architecture follows directly. Frontend in vanilla TS. Netlify Functions in TypeScript for the request flow. Rust for the strategy library, compiled to WebAssembly, called from inside the background function. Each language is in the layer where it adds the most value, and the cost of working across the boundary (the WASM bridge, serde-wasm-bindgen for JSON marshalling, the StrategyKind dispatch enum) is paid once and benefits every strategy that ships.

Lesson 03 · Ownership

Ownership and borrowing in numerical code.

The single most-cited reason Rust feels different from other languages is the ownership system. The clearest way to learn it is to read code that uses it sensibly. This site's &[DailyBar] parameter shape is a working example of the right default.

The three rules

Rust's ownership system is three sentences long:

  1. Each value has a single owner.
  2. When the owner goes out of scope, the value is dropped.
  3. A value can be borrowed (shared or mutable) but only one mutable borrow exists at a time, and no shared borrow can coexist with a mutable one.

Every Rust program is an application of these three rules to concrete types. The compiler checks them at compile time; if they're violated, the code doesn't build.

What "move" actually means

When you write let w = v; for a type that doesn't implement Copy (like Vec<T> or String), the value moves from v into w. v can no longer be used; the compiler will reject any subsequent reference to it. 

let v = vec![1, 2, 3];
let w = v;        // v moved into w
println!("{:?}", v);  // ERROR: value moved here

For primitive types that do implement Copy (i8 through i128, f32, f64, bool, char), assignment makes a bitwise copy and the original stays valid. That's why let x = 5; let y = x; works fine.

Borrowing: the right default for read-only access

Most code doesn't actually need to take ownership of its inputs. A function that walks a vector and returns some derived value (a sum, a mean, a max) just needs to read the vector. Borrowing with &Vec<T> (or more idiomatically &[T]) lets the function read the data without claiming ownership.

This site's strategy functions all follow this pattern. Every positions() function takes bars: &[DailyBar]. The strategy walks the slice, computes a position vector, returns it. The caller keeps the bar series and can hand it to another strategy without cloning.

Why &[T] and not &Vec<T>?

Slices (&[T]) are more general than references to vectors (&Vec<T>). A function that takes &[T] can be called with a Vec (which auto-derefs to a slice), an array slice (&arr[..]), or any other slice-like type. A function that takes &Vec<T> can only be called with an actual Vec.

The convention is: take &[T] in function parameters when you only need to read the data. Take &mut Vec<T> if you need to push to it. Take Vec<T> if you need to own it (e.g., to store it somewhere). The default should be the most general read-only form.

The borrow checker in practice

Rust's borrow checker enforces the rules at compile time. Two patterns trip up new Rust programmers most often:

1. Iterating and mutating simultaneously

let mut v = vec![1, 2, 3];
for x in &v {
    if *x == 2 {
        v.push(4);   // ERROR: cannot borrow v as mutable
    }                // because it is also borrowed as immutable
}

The fix is to collect the indices you want to modify in a first pass, then modify in a second pass after the immutable borrow ends. Or restructure to avoid the simultaneous read+modify.

2. Returning a reference to a local

fn bad() -> &[i32] {
    let v = vec![1, 2, 3];
    &v[..]    // ERROR: returns reference to local v
}              // v is dropped here, reference would dangle

The fix is to return the owned Vec<i32> instead. The borrow checker catches the dangling-reference bug at compile time; in C++ this would silently produce undefined behavior.

What this buys you in practice

Three things you stop worrying about:

  • Use-after-free. The borrow checker won't let you reference something whose owner has been dropped.
  • Double-free. Each value has exactly one owner; when that owner goes out of scope, the value is dropped exactly once.
  • Iterator invalidation. Mutating a collection while iterating it is a compile error, not a runtime crash.
The mental shift. Move ownership only when you mean to. Borrow shared when you only need to read. Borrow mutable when you need to write, but only one at a time. The compiler enforces these by default; the friction in the first few weeks of Rust is the compiler teaching you to be explicit about which mode you're in.
